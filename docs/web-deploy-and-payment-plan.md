# 网站版部署与收费接入方案

## 1. 当前网站版位置

网站版代码在：

```text
web/
```

本地运行：

```bash
cd web
npm install
npm run dev
```

本地访问：

```text
http://127.0.0.1:5173
```

生产构建：

```bash
npm run build
```

构建产物：

```text
web/dist
```

## 2. 当前版本能力

当前 Web/H5 版已经包含：

- 用户登录 / 注册：姓名 + 手机号。
- 管理员登录：默认密码 `admin123`。
- 用户端三栏：首页练习、题库中心、我的。
- 练习模式：顺序练习、随机练习、专项练习、章节练习、模拟考试、错题、收藏。
- 即时判题：单选和判断选择后立刻反馈，多选点确认后反馈。
- 考试逻辑：整套试卷提交后统一批卷，满分 100。
- 题库中心：管理员上传发布，用户加入自己的题库后练习。
- 管理后台：题库管理、Word/TXT 导入、考试模板、激活码、用户授权、订单、备份、数据统计。
- 收费预留：付费题库、套餐、激活码、订单、手动授权。

当前版本是静态前端演示版，数据保存在浏览器 `localStorage`。它适合免费部署、给客户试用、验证交互流程；不适合直接做正式多用户收费运营。

## 3. 免费部署方式一：Cloudflare Pages

优点：

- 免费额度较高。
- 适合部署 Vite 静态网站。
- 后续可以继续接 Cloudflare Workers、D1、R2。

步骤：

1. 把项目推送到 GitHub。
2. 打开 Cloudflare Pages。
3. 选择 `Create a project`。
4. 连接 GitHub 仓库。
5. Framework preset 选择 `Vite`。
6. Root directory 填：

```text
web
```

7. Build command 填：

```bash
npm run build
```

8. Build output directory 填：

```text
dist
```

9. 点击部署。

部署完成后会得到类似：

```text
https://question-bank.pages.dev
```

学生可以直接打开这个网址使用。

## 4. 免费部署方式二：Vercel

优点：

- 操作简单。
- GitHub 更新后可自动重新部署。
- 适合快速展示。

步骤：

1. 打开 Vercel。
2. 选择 `Import Git Repository`。
3. 选择你的 GitHub 仓库。
4. Root Directory 选择：

```text
web
```

5. Framework Preset 选择 `Vite`。
6. Build Command 填：

```bash
npm run build
```

7. Output Directory 填：

```text
dist
```

8. 点击 Deploy。

部署完成后会得到类似：

```text
https://question-bank.vercel.app
```

## 5. 静态版限制

静态版的优点是部署快、成本低、不需要服务器。

限制是：

- 每个浏览器的数据互不共享。
- 用户换设备后看不到原来的练习记录。
- 管理员导入的题库只保存在当前浏览器。
- 真实支付回调无法只靠前端完成。
- 管理员密码写在前端，不适合正式运营。
- 无法可靠防止用户绕过前端限制。

因此，当前版本可以作为演示版和早期试用版。正式收费前需要接后端。

## 6. 收费落地路线

### 阶段一：激活码收费

这是最快落地方式。

流程：

1. 管理员在后台生成激活码。
2. 用户通过微信、支付宝、线下转账等方式付款。
3. 管理员把激活码发给用户。
4. 用户在“我的”页面输入激活码。
5. 系统开通会员或单题库授权。

优点：

- 不需要马上申请复杂支付接口。
- 适合早期验证定价。
- 可以用于机构批量售卖。

缺点：

- 付款和发码需要人工处理。
- 正式规模化后需要订单后台和自动发码。

### 阶段二：收款码 + 人工审核

流程：

1. 用户点击购买套餐。
2. 系统生成预留订单。
3. 页面展示收款码或付款说明。
4. 用户付款后提交订单号或截图。
5. 管理员后台手动标记订单已支付。
6. 系统自动开通授权。

这个方式仍然不需要复杂支付接口，但需要后端保存订单。

### 阶段三：正式支付接口

正式支付流程：

1. 用户点击购买题库或会员。
2. 前端请求后端创建订单。
3. 后端调用微信支付、支付宝或第三方支付接口。
4. 用户支付。
5. 支付平台回调后端。
6. 后端校验签名和金额。
7. 后端更新订单为已支付。
8. 后端开通题库授权或会员。
9. 前端刷新权限。

正式支付一定需要后端，不能只靠静态网页完成。

## 7. 推荐后端方案

### 方案 A：Supabase

适合快速正式化：

- 用户表
- 题库表
- 题目表
- 做题记录
- 错题和收藏
- 订单
- 授权
- 激活码

优点是开发速度快，免费额度够早期使用。

### 方案 B：Cloudflare Pages + Workers + D1

适合继续沿用 Cloudflare：

- Pages 部署前端。
- Workers 做 API。
- D1 做数据库。
- R2 保存 Word 文件。

优点是整体成本低，和静态部署衔接自然。

### 方案 C：自有服务器

适合正式商业化：

- Node.js / Java / Python 后端。
- MySQL 或 PostgreSQL。
- Word 解析服务。
- 支付回调服务。
- 管理员后台 API。

优点是最可控，缺点是运维和备案成本更高。

## 8. 正式上线前清单

- 域名。
- ICP 备案：如果使用中国大陆服务器或大陆 CDN。
- 用户协议。
- 隐私政策。
- 退款规则。
- 客服联系方式。
- 管理员权限系统。
- 数据备份方案。
- 支付订单系统。
- 支付状态校验。
- 题库版权和授权来源。
- 日志与异常监控。

## 9. 建议执行顺序

1. 先把当前 `web/` 部署到 Cloudflare Pages 或 Vercel。
2. 找少量真实用户试用，验证刷题和考试体验。
3. 用激活码模式验证收费意愿和定价。
4. 接 Supabase 或 Cloudflare D1，把数据从 `localStorage` 迁移到云端。
5. 加入正式订单和支付流程。
6. 再考虑小程序、H5、App 多端统一账号。
