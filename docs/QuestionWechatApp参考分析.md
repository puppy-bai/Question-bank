## 2. 项目基本判断

这是一个较早期的微信原生小程序答题项目，代码主体是小程序端：

- 页面：约 17 个业务页面。
- 后端：使用 `utils/Bmob-1.6.7.min.js`，即 Bmob 后端服务。
- 数据：通过 Bmob 表和外部 JSON 题目地址获取。
- 仓库内未发现 PHP、SQL、Java 或其他后台源码。
- README 中提到“有前后端完整源码和数据库”“PHP/MySQL 运行环境”，但当前 GitHub 仓库实际只包含小程序端代码和静态资源。

核心页面：

| 页面 | 作用 |
| --- | --- |
| `pages/index` | 首页入口 |
| `pages/category` | 题库/分类选择 |
| `pages/learn` | 练习模式 |
| `pages/exam` | 模拟考试 |
| `pages/moniq` | 答题模板组件 |
| `pages/wrong` | 错题入口 |
| `pages/errorStar` | 错题练习 |
| `pages/rank` | 排名 |

## 4. 可取之处

### 4.1 答题模式 / 背题模式

`pages/learn/index.wxml` 提供了两个模式：

- 答题模式：作答后显示正误和解析。
- 背题模式：直接展示正确答案和解析。

这个很适合我们的学生端。第一版可以在练习页加一个模式切换，学生复习时用背题模式效率更高。

建议迁移优先级：高。

### 4.2 答题卡

练习页底部有答题卡：

- 正确数量。
- 错误数量。
- 当前题号 / 总题数。
- 每题状态颜色：未答、当前、正确、错误。
- 点击题号跳转。

这个体验比我们当前“下一题”简单模式强很多，尤其适合学校内部刷题。

建议迁移优先级：高。

### 4.3 断点续答

练习和考试都使用本地缓存保存进度：

- `wx.setStorageSync(cateid, saveStorageData)`
- `wx.setStorageSync('exam' + cateid, saveStorageData)`

保存内容包括题目列表、正确数、错误数、当前题号、剩余时间等。

我们不建议照搬本地缓存作为唯一状态，但可以借鉴思路：本地即时保存 + 云端 `study_sessions` 定期保存。这样断网或退出后体验更稳。

建议迁移优先级：高。

### 4.4 模拟考试

`pages/exam` 包含：

- 随机抽题。
- 倒计时。
- 自动保存考试进度。
- 交卷确认。
- 成绩保存。
- 跳转结果页。

我们当前还没有考试模式，这一块可以作为第二阶段的重要功能参考。

建议迁移优先级：中高。

### 4.5 错题本入口设计

它把错题按题库分类保存，先展示错题所属题库，再进入错题练习页。

我们当前错题页直接列错题。后续可以改成：

1. 按题库聚合错题。
2. 进入某题库错题练习。
3. 支持移除错题。

建议迁移优先级：中。

### 4.6 排名和记录

它有：

- `history` 成绩记录。
- `rank` 按某题库最高分排名。
- `record` 查看个人历史记录。

学校内部使用时，排名不一定第一版需要，但“学生历史记录”和“老师按班级看成绩”很有价值。

建议迁移优先级：中。

### 4.7 用户信息审核

`register` 里有公司、部门、真实姓名、审核状态。README 也提到“信息审核，只有通过审核的人员才可进入答题”。

我们可以映射成学校场景：

- 姓名。
- 学号。
- 班级。
- 角色。
- 审核状态。

建议迁移优先级：中高。

## 5. 不建议直接照搬的地方

### 5.1 Bmob 后端

参考项目依赖 Bmob SDK 和 Bmob 表。我们已经选择微信云开发 / CloudBase，且需要学校内部权限、老师数据、Word 导入。直接迁移 Bmob 会让系统架构变复杂。

结论：只借鉴交互和数据思路，不搬后端 SDK。

### 5.2 答案暴露在前端

参考项目获取题目时会把 `answer` 一起加载到前端，再前端判题。对普通练习影响不大，但对学校内部考试或老师统计来说不够严谨。

我们当前 `getQuestions` 不返回答案，`submitAnswer` 云函数判题，这条更安全，应该保留。

### 5.3 错题只存在本地缓存

参考项目错题主要用 `wx.setStorageSync` 保存。换设备或清缓存会丢失，也不利于老师查看。

我们当前写入 `wrong_questions` 云数据库，更适合学校内部管理。

### 5.4 代码年代较旧

可见问题：

- 使用旧 `wx.getUserInfo` 授权路径。
- 存在大量 `console.log`。
- 多处逻辑强耦合页面状态。
- 多选题模板里硬编码到 10 个选项，WXML 判断很长。
- `project.private.config.json` 这类个人配置不应提交。

结论：不适合整体搬代码，只适合拆思路重写。

## 6. 对我们项目的迁移建议

### 第一优先级

1. 练习页增加“答题模式 / 背题模式”。
2. 增加底部答题卡：正确、错误、当前进度、题号跳转。
3. 增加本地断点续答，同时继续写云端 `study_sessions`。
4. 练习结束页：正确数、错误数、未答数、用时。

当前迁移状态：

- 已完成“答题模式 / 背题模式”。
- 已完成底部答题卡。
- 已完成本地断点续答和清空本地记录。
- 已完成练习结果页：正确数、错误数、未答数、正确率、用时、再练一轮、查看错题、返回题库。

### 第二优先级

1. 模拟考试模式：随机抽题、倒计时、交卷。
2. 个人答题记录页。
3. 错题按题库聚合。
4. 学生账号审核状态。

### 第三优先级

1. 排名。
2. 签到/积分。
3. 反馈回复。

## 7. 结论

这个项目的最大价值不在架构，而在“成熟刷题产品的交互样式”：

- 答题卡。
- 背题模式。
- 断点续答。
- 模拟考试。
- 结果页。
- 错题专项练习。

它的后端、权限、安全模型不适合直接用于我们当前学校内部题库小程序。我们应当继续使用现有 CloudBase 架构，把它的交互体验拆出来，用我们自己的数据模型和云函数重做。
